# AGENTS.md

Guidance for AI coding agents working in this repository.

## Project

FastAPI backend for managing family recipes ("Hungry Ninja"). Python 3.13, managed with `uv`. Persistence via SQLModel (SQLAlchemy) against a database configured by `DATABASE_URL` (SQLite file `hungry_ninja.db` in this environment). Auth is Auth0 (`fastapi-auth0`). Recipe tag/substitution suggestions are generated by `pydantic-ai` agents backed by an LLM (`LLM_MODEL` env var, currently Gemini via `GOOGLE_API_KEY`). Observability via `logfire` (FastAPI, SQLAlchemy, and pydantic-ai instrumentation are all wired up in `app/main.py` / `app/services/recipe_service.py`).

## Commands

Dependencies are managed with `uv` (see `uv.lock`). Prefix commands with `uv run` or activate `.venv` first.

- Install/sync deps: `uv sync`
- Run the API locally: `uv run fastapi dev app/main.py`
- Run all tests: `uv run pytest`
- Run a single test file: `uv run pytest tests/test_example.py`
- Run a single test: `uv run pytest tests/test_example.py::test_example`
- Test config (`pyproject.toml`) auto-enables coverage (`--cov=app --cov-report=term-missing`) and `asyncio_mode = "auto"`, so `async def test_*` functions work without `@pytest.mark.asyncio`.

There is no configured linter/formatter in `pyproject.toml`; don't assume ruff/black/mypy are wired in unless you add them.

## Environment

Required env vars (see `.env`, loaded via `load_dotenv()` in `app/main.py`): `DATABASE_URL`, `LLM_MODEL`, `GOOGLE_API_KEY`, `AUTH0_CLIENT_ID`, `AUTH0_DOMAIN`, `AUTH0_AUDIENCE`.

## Architecture

Layered structure, each layer with its own package and an `__init__.py` that re-exports the public surface (import from the package, not the submodule, e.g. `from app.data import Recipe`, `from app.services import RecipeServiceDep`):

- `app/data/` — SQLModel table models (`table=True`) plus DB wiring. `app/data/__init__.py` creates the `engine` from `DATABASE_URL`, exposes `init_db()` (called from the FastAPI `lifespan`) and `get_session()` / `SessionDep` (a `Depends`-annotated `Session` for use in services). All entity models inherit `AuditableModel` (`app/data/auditable_model.py`), which supplies `id`, `created_at/by`, `updated_at/by`, and `marked_for_deletion`.
- `app/models/` — Pydantic DTOs used at the API boundary (request/response bodies), distinct from the `app/data` table models. Mapping between the two happens explicitly in `app/mapping/` (e.g. `recipe_mapping.to_recipe_dto`) rather than serializing table models directly.
- `app/services/` — business logic. Services are FastAPI-dependency-injectable classes (e.g. `RecipeService.__init__(self, user: AuthUserDep, session: SessionDep, family_service: FamilyService = Depends(FamilyService))`), exposed to routers as `Annotated[..., Depends(...)]` aliases in `app/services/__init__.py` (`RecipeServiceDep`, `FamilyServiceDep`). Domain errors are plain exceptions in `app/services/exceptions.py` (`NotFoundException`, `UnauthorizedException`, `NoFamilyException`, `DuplicateException`) — routers catch these and translate to `HTTPException`s.
- `app/routes/` — `APIRouter`s, one per resource, each gated by `Depends(auth.implicit_scheme)`. Routers are thin: they call a service method, catch the service's domain exceptions, and map them to HTTP status codes.
- `app/auth/` — Auth0 integration (`app/auth/__init__.py`); `auth.implicit_scheme` guards routes, `AuthUserDep` injects the authenticated `Auth0User` into services.

### Data model

Recipes belong to a `Family` (multi-tenancy boundary — most queries filter by the caller's family, resolved via `FamilyService.get_family_by_auth_id()` from the authenticated user's `FamilyMember.auth_id`). A `Recipe` has one or more `RecipePartGroup`s (e.g. "dough", "filling") — this is how the API represents recipes with distinct components. Each group has ordered `Instruction`s and `RecipeIngredient`s; a `RecipeIngredient` points at either an `Ingredient` or another `Recipe` (`linked_recipe_id`, for sub-recipes/components). Tags (`Tag`, `RecipeTag`) are family-scoped. `SuggestedTagHistory` tracks per-family-member accept/reject counts for LLM-suggested tags, feeding future suggestions (see `RecipeService._get_top_tags` / `_get_bottom_tags`). See `er.mmd` for the full ER diagram.

### LLM-suggested tags/substitutions

`RecipeService` builds three `pydantic_ai.Agent`s at construction time (tag suggestion, tag suggestion with tool access to top/bottom tag history, and ingredient substitution), all with `output_type=list[SuggestionDTO]`. Suggested tags are persisted to `SuggestedTagHistory` via `_save_tag_suggestions`, matching against existing tags for the same family member before creating new history rows. `ModelHTTPError` from `pydantic_ai` is caught in the routers and surfaced as a 503.
